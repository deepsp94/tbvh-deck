TBVH

To Be Verifiably Honest

A trustless marketplace for buying and selling information.
AI agents negotiate inside a TEE. Cryptographic proofs replace trust.
On-chain escrow enforces the deal.

Production implementation of the NDAI paper by Stephenson, Miller, Sun, Annem, Parikh (2025)

The Problem

Arrow's Information Paradox

→
You can't evaluate information without seeing it. Once you've seen it, why pay? This is Arrow's Information Paradox (1962), unsolved for 60 years.
→
From the NDAI paper: John Harrison built the Marine Chronometer and revealed it to claim his reward. Bureaucrats had no incentive to pay once they'd seen it. It took 40 years and a king's intervention to get paid.
→
The NDAI paper proves that TEEs + AI agents eliminate this hold-up problem. Under the TEE arrangement, the unique equilibrium is full disclosure at an efficient price. Both parties are strictly better off.
→
TBVH is the production system built on this theory.
Architecture

How TBVH Works

01
Post a request. The buyer describes the information they need and sets a budget in USDC. Sellers commit their information and optional cryptographic proof.
02
Agents negotiate inside the TEE. Each side has an AI agent. They negotiate inside a Phala Cloud CVM (Intel TDX), hardware-isolated memory that nobody can read. Neither human sees the conversation. The buyer only learns: accept or reject, at what price.
03
Settle on-chain. If accepted, the buyer deposits USDC into an escrow contract. The TEE verifies the deposit, signs the outcome (EIP-712), and reveals the seller's information. The seller claims payment with the TEE signature. If rejected, information is wiped and the buyer never sees it.
Proofs

Cryptographic Proofs

→
The NDAI paper acknowledges that agents make errors (Section 4). How do you increase the buyer agent's confidence in the seller's claims?
→
TBVH's answer: verifiable proof types that the TEE validates independently. Not claims from the seller, but system-verified evidence.
→
This directly reduces the error term ε from the paper's agent model.

Today: DKIM Email Verification

Sellers attach .eml files. The TEE verifies the cryptographic DKIM signature, confirming the email is genuine and unmodified.

Both agents see the verified content. After the deal, email content is wiped from the TEE.

Coming: zkTLS + more

Extensible proof_type field designed from day one. zkTLS for web content verification, document attestation, and more.

Settlement

On-Chain Escrow

→
TBVHEscrow contract on Base Sepolia. Holds USDC deposits keyed by negotiation ID.
→
EIP-712 typed signatures. The escrow only trusts the TEE signer. Key is derived from hardware, bound to the CVM identity.
→
Attestation-verifiable. Anyone can confirm the exact code running in the TEE.

release()

Seller claims payment with TEE-signed ACCEPT outcome

refundWithSig()

Buyer gets immediate refund with TEE-signed REJECT outcome

refund()

7-day timeout fallback. No signature needed. Funds are never locked permanently.

CLI

tbvh-cli: Agents Talking to Agents

→
No frontend needed. Full CLI client using bash, curl, and Foundry's cast. Designed for maximum security because frontends can be compromised.
→
Programmatic access. Other AI agents can use tbvh-cli as a tool, enabling agents buying information from agents autonomously.
→
Complete coverage. Auth (SIWE), create requests, commit as seller (text + email proof), stream negotiations, escrow deposit/release/refund.
./auth.sh ./buyer.sh create \ "Snowflake pricing" \ "Enterprise per-credit rates" \ 50 ./stream.sh buyer $INSTANCE_ID ./seller.sh commit-email \ $INSTANCE_ID \ "I have the pricing" \ ./contract.eml ./escrow.sh deposit $NEG_ID 35 ./escrow.sh release $NEG_ID

Learnings

From Paper to Production

Key derivation. Phala's TEE signer is bound to the CVM's app_id, not the Docker image hash. In-place code updates don't rotate keys. Only full reprovisioning does. The DStack docs and practical behavior diverge here.
Prompt engineering is critical. Without explicit skepticism instructions and adversarial framing (the seller has a financial incentive to fabricate), the buyer agent accepts fabricated information. Structured XML prompts with proof verification status were needed to get reliable behavior.
The paper's error model is real. Section 4 of the NDAI paper models agent errors and shows that budget caps and acceptance thresholds preserve efficiency gains. In practice, budget caps (max_payment) and cryptographic proof verification (DKIM) are the exact mitigations we implemented.
Roadmap

What's Next

zkTLS proof type. Verify web content (API responses, web pages) without trusting the seller. Extends the proof system beyond email.
Production escrow on mainnet. Deploy to Base mainnet with real USDC. The same contracts, real money.
